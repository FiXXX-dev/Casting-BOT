# Supabase-схема профиля · обоснование

Схема покрывает оба раздела мини-аппа из `index.html`:
«Что ищу» (фильтры подбора) и «Моя анкета» (портфолио).
SQL — в `supabase/migrations/0001_init_profiles.sql`.

## Карта таблиц

| Таблица | Раздел UI | Связь | Кто читает |
|---|---|---|---|
| `actors` | — (якорь) | 1 запись = 1 Telegram-пользователь | бот по `telegram_user_id` |
| `search_filters` | «Что ищу» | 1:1 с `actors` | движок мэтчинга (часто) |
| `portfolios` | «Моя анкета» | 1:1 с `actors` | автозаполнение откликов (редко) |

### Почему три таблицы, а не одна

1:1 можно было свернуть в одну широкую таблицу, но раздельно лучше, потому что:

- **Разные паттерны чтения.** Мэтчинг гоняет только `search_filters` по всем активным актёрам — это горячий путь. Портфолио нужно эпизодически (когда формируем отклик). Узкие строки фильтров = меньше I/O в основном цикле.
- **Карта 1:1 с UI.** Верхний переключатель «Что ищу / Моя анкета» = две таблицы. Фронт грузит и сохраняет ровно тот блок, который открыт.
- **Изоляция миграций.** Поля портфолио меняются чаще (контакты, навыки), фильтры — стабильнее. Меняем независимо.

Свернуть в одну таблицу — тоже валидно, если хочется минимум join'ов. Это легко обратимо; начинаем с разделения.

## Связь telegram_user_id ↔ профиль

- `actors.telegram_user_id` — `BIGINT NOT NULL UNIQUE`. Это **натуральный ключ от бота**: пришёл апдейт → `select … where telegram_user_id = ?`. `BIGINT`, потому что Telegram-id уже выходят за пределы `int4` (>2.1 млрд).
- Внутренний PK — `id uuid`. Дочерние таблицы ссылаются на `actor_id uuid`, а не на telegram-id. Зачем развязка:
  - telegram-id не протекает во все FK;
  - если завтра добавится второй канал (web, другой мессенджер), он станет ещё одной строкой-идентичностью, а профиль не переклеивать.
- `ON DELETE CASCADE`: удалили актёра — фильтры и портфолио уходят сами.

> Можно было сделать PK прямо `telegram_user_id BIGINT` и не плодить uuid — для одного канала разницы нет. Выбрал uuid ради развязки на будущее; это решение дёшево поменять.

## Колонка или JSONB или массив — правило

| Тип данных | Хранение | Почему |
|---|---|---|
| Одиночный скаляр/булево, по которому фильтруем или который показываем | **колонка** (+ `CHECK`) | типизация, индексы, читаемость |
| Плоский мультиселект, по которому делаем set-операции | **`text[]`** (+ GIN) | нативный `&&` overlap в мэтчинге без join'ов |
| Вложенная/группированная структура, которую читает LLM | **`jsonb`** | структура растёт без миграций, отдаётся в gpt-4o-mini как есть |

Принцип: **по чему фильтруем в SQL — колонки и массивы; что просто храним и скармливаем LLM — JSONB.**

### Разбор по полям

**`search_filters` («Что ищу»):**

- `gender` — `text CHECK in ('male','female')`. Single-выбор, фильтруется. `CHECK`, а не enum: enum в Postgres тяжело расширять (`ALTER TYPE` + блокировки), а `CHECK` правится одним стейтментом.
- `play_age_min` / `play_age_max` — две `smallint` колонки (вилка «от-до»). `CHECK` на диапазон 1..99 и `play_age_min <= play_age_max`. Двумя колонками, потому что фронт показывает два поля; для мэтчинга «возраст кастинга попадает в вилку» это `casting_age between min and max`.
  - Альтернатива: `int4range` + GiST + оператор `@>`. Элегантно для пересечений, но менее очевидно фронту. Оставил две колонки; перейти на range легко.
- `search_city` — `text` (`tashkent`/`moscow`/`any`). Не строгий enum: города будут добавляться, не хочется ловить миграцию на каждый новый.
- `project_types` — **`text[]`** (`film`/`ad`/`model`/`voiceover`). Это главный мультиселект мэтчинга → GIN-индекс и запрос `project_types && array['film']`.
- `min_rate` — `integer` (сумы), `NULL` = «показывать всё» (в UI пустое поле = без ограничения).
- `exclusions` — **`jsonb`** `{"specifics":[…], "typecasts":[…]}`. Две именованные группы стоп-слов («Специфика проекта» / «Типажи мимо»), значения — открытый список (актёр допишет своё). JSONB держит обе группы и произвольные строки; их же читает LLM-фильтр.
- `reach` — `text CHECK in ('broad','exact')`, дефолт `broad`. Верхний переключатель охвата.

**`portfolios` («Моя анкета»):**

- `full_name`, `age` (реальный, отдельно от игрового), `city`, `eye_color` — простые колонки.
- `travel_ready` — `boolean` (switch «Готов к командировкам»).
- `employment` — `text CHECK in ('self_employed','ip','individual')` («Как оформлен»).
- `has_experience` — `boolean` («Снимался раньше: Да / Пока нет»).
- `education` — `text CHECK in ('university','courses','university_and_courses','none')` (single-list).
- `distinguishing_features` — **`text[]`** (Татуировки/Пирсинг/…). Плоский список без групп → массива достаточно.
- `skills` — **`jsonb`** `{"sport":[],"dance":[],"vocal":[],"instruments":[],"languages":[]}`. Сгруппированный мультиселект из пяти категорий. JSONB: добавить категорию = не трогать схему, и весь объект уходит в LLM при подборе. GIN-индекс на случай поиска «кто говорит по-узбекски».
- контакты `portfolio_url`, `showreel_url`, `phone`, `contact_telegram`, `email` — отдельные `text`-колонки (структурированы, идут в автозаполнение отклика).

### Канонические ключи vs отображаемые подписи

Фиксированные перечисления (`gender`, `project_types`, `employment`, `education`, `reach`, `search_city`) храним **слугами на латинице**, русские подписи живут на фронте. Так данные не зависят от формулировок в UI.

Открытые наборы (`skills`, `exclusions`, `distinguishing_features`) храним **как отображаемые строки** — они и так уходят в gpt-4o-mini, где русский текст осмысленнее слуга, и допускают пользовательские значения.

## RLS и кто пишет

- На всех таблицах `ENABLE ROW LEVEL SECURITY`.
- **Писатель — бот/бэкенд через `service_role`-ключ**, он обходит RLS. Для текущей архитектуры (мини-апп → `sendData` → бот → Supabase) этого достаточно, политики не нужны.
- Если мини-апп пойдёт в Supabase **напрямую** (supabase-js из WebApp), потребуется: edge-функция проверяет Telegram `initData` (подпись по токену бота) и выдаёт JWT с `telegram_user_id` в claim; политики пускают пользователя только к своей строке. Заготовка политик — закомментирована в миграции, чтобы не включать половину механизма.

## updated_at

Один триггер `set_updated_at()` на три таблицы — без дублирования логики и без доверия к тому, что клиент проставит время.

## Открытые вопросы к тебе

1. Мини-апп пишет в Supabase напрямую или всё через бота? От этого зависит, доделывать ли RLS-политики.
2. Города — оставляем свободным `text` или нужен справочник `cities` (тогда отдельная таблица + FK)?
3. Игровой возраст — две колонки или перейти на `int4range` ради пересечений в мэтчинге?
